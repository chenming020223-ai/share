# 公网分享部署说明

日期：2026-05-27  
目标：向受邀体验者发送网页链接，由服务器统一获取 API-Football 数据并生成中文分析报告。

## 一、交付模式

当前推荐采用“受控分享版”，而不是匿名公开站点：

- 访客通过公网网址访问，例如 Render 提供的 `https://...onrender.com` 地址。
- 首次进入由浏览器要求输入共享用户名与访问口令。
- API-Football 密钥只配置在托管平台的服务器环境变量中，网页不显示密钥输入框，也不接受访客覆盖密钥。
- 所有持有口令的受邀用户共享同一份预测历史和模型验收状态，适合小范围评审和演示。

若后续需要完全公开给不认识的访客使用，应增加账号体系、调用额度控制、用户数据隔离和操作审计，不应直接开放当前共享数据模式。

## 二、为什么不能直接分享本地链接

- `127.0.0.1` 只能在当前电脑访问。
- 当前 API-Football 密钥已用于本地试验；公网发布前应在 API 平台重新生成密钥，旧密钥停用。
- 线上服务必须将数据库放到持久存储，否则重启或重新部署后，历史预测与回测样本会丢失。

## 三、已完成的发布适配

- 新增 `render.yaml`，用于创建公网 Python 服务、健康检查和持久磁盘。
- 新增 `WORLDCUP_PUBLIC_MODE=true` 分享模式：API 密钥由服务器托管，访客界面自动隐藏密钥框。
- 新增 `WORLDCUP_ACCESS_USER` 与 `WORLDCUP_ACCESS_PASSWORD`：启用共享口令保护。
- 服务支持托管平台提供的 `PORT` 端口和 `/healthz` 健康检查。
- 本地模式保持原有使用方式，不影响电脑上继续研究。

## 四、Render 发布步骤

1. 将项目上传到甲方控制的 GitHub 私有仓库。
2. 在 Render 创建 Blueprint，并选择仓库中的 `render.yaml`。
3. 填写两个不进入代码库的秘密变量：
   - `API_FOOTBALL_KEY`：新生成的 API-Football 密钥。
   - `WORLDCUP_ACCESS_PASSWORD`：提供给受邀体验者的访问口令。
4. Render 完成部署后会提供公网网址。
5. 用电脑和手机各进行一次预测、Excel/PDF 导出和刷新后历史保存验收。

`render.yaml` 配置了持久磁盘，因此该路线需要 Render 的付费 Web Service；这是因为 SQLite 历史记录和模型样本不能依赖临时文件系统。

## 五、甲方需要提供

- 托管选择：建议确认采用 Render。
- 一个可连接部署的 GitHub 私有仓库，或允许协助创建仓库并发布。
- 新的 API-Football 密钥；不要继续使用曾在交流中出现过的旧密钥。
- 希望设置的共享访问用户名和访问口令。
- 是否允许受邀用户看到共同的“历史报告”和“模型审计”数据。

## 六、验收标准

- 通过公网 HTTPS 链接访问，并在输入正确口令后打开界面。
- 分享模式页面不显示 `API 密钥` 输入框。
- 可选择真实比赛并生成预测，队名规则为“已核定中文优先、否则显示 API 原名”。
- 可导出中文 Excel 和 PDF。
- 服务重新部署或重启后，历史预测仍保留。
- 错误口令不能访问页面或 API。
