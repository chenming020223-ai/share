# 世界杯预测模型数据规范 V1.0

状态：第一版工程基线  
预测口径：仅计算 90 分钟赛果  
适用范围：网页 API 模式、模拟舱、中文 Excel/PDF 报告、后续回测数据集

## 1. 基本原则

1. 比赛、球队和赛事以 API-Football ID 作为关联键；中文名称仅用于显示和报告。
   API-Football 不提供中文官方名字段，项目对已核定名称优先展示受控中文；对未核定名称直接展示 API 英文原名，供甲方识别、核定和审计。
2. 每次预测抓取双方最近 10 场已完成比赛，使用 90 分钟比分形成近期强度。
3. 双方各至少有 5 场有效近期比赛，模拟舱才允许继续评估信号。
4. 正式 API 模式的赔率来源固定为 `Pinnacle`，不混合不同庄家公司。
5. 盘口必须是全场胜平负、全场亚洲让球或全场进球大小球。
6. 缺失、异常、存在重大模型分歧或未具备经验证 `pfinal` 的数据只用于展示和复核，不产生模拟买入信号。

## 2. 比赛身份字段

| 字段 | 必须 | 说明 |
|---|---:|---|
| `fixture_id` | 是 | API-Football 比赛唯一 ID |
| `league_id` | 是 | 赛事 ID |
| `league_name_cn` | 是 | 官方中文赛事名，用于报告 |
| `home_team_id` / `away_team_id` | 是 | 球队唯一 ID |
| `home_team_name_cn` / `away_team_name_cn` | 是 | 官方中文队名，用于展示 |
| `kickoff_beijing` | 是 | 北京时间 |
| `venue` | 建议 | 比赛场地 |
| `neutral_site` | 是 | 世界杯正赛按中立场；非世界杯赛事按实际主客场 |

## 3. 球队近期数据

| 规则 | 值 |
|---|---:|
| 请求窗口 | 最近 10 场 |
| 有效比赛最低数量 | 每队至少 5 场 |
| 有效结果口径 | 90 分钟完整比分 |
| 延长赛/点球比赛 | 只取全场 90 分钟比分，不取晋级结果 |

当前基础版用于模型计算的近期字段：

| 字段 | 用途 |
|---|---|
| `goals_for` | 近期进攻表现 |
| `goals_against` | 近期防守表现 |
| `points` | 近期结果强度 |
| `venue` | 主客场审计 |
| `date` | 数据时点审计 |

后续增强字段：`xG`、`xGA`、射门、射正、红牌、点球、预计首发与伤停。增强字段缺失时不得伪造为数值。

## 4. 统一庄家与盘口字段

正式模拟庄家固定为 `Pinnacle`。

| 市场 | 允许名称 | 必须条件 |
|---|---|---|
| 胜平负 | 全场 `Match Winner` / `1X2` | 主胜、平局、客胜三项齐全 |
| 大小球 | 全场 `Goals Over/Under` | 同一盘口线的大、小赔率齐全 |
| 让球 | 全场 `Asian Handicap` | 同一盘口线的主、客赔率齐全 |

禁止纳入：

- 半场胜平负或半场让球；
- 卡牌、角球、球员等其他大小球；
- 不同庄家之间拼接盘口；
- 多庄家平均赔率作为模拟下注赔率。

每条赔率数据至少保存：

| 字段 | 说明 |
|---|---|
| `bookmaker` | 必须为 `Pinnacle` |
| `market_type` | 全场市场种类 |
| `line` | 大小球或让球盘口值 |
| `odds` | 对应十进制赔率 |
| `captured_at` | API 返回的赔率更新时间/抓取时间 |

当接口同时返回多个让球线或大小球线时，程序以同一庄家、同一全场市场中两侧赔率最接近平衡的一组作为主盘口，不能固定选择零盘或固定选择 `2.5`。

## 5. 概率对象与模拟信号硬门槛

概率身份固定为：

| 对象 | 当前来源 | 用途 |
|---|---|---|
| `pbase` | 基础进球/比分模型概率 | 研究与校准输入，不得直接产生正式信号 |
| `qmkt` | Pinnacle 指定盘口去水概率 | 市场基准 |
| `pshr` | 已实现时间切分收缩审计流程，待足量样本拟合/验收 | 正式链路中间层 |
| `pfinal` | 待验证的执行概率 | 未来正式 EV 唯一概率输入 |

当前展示的融合概率只供界面比较，不等同于 `pfinal`。

当前 `pshr` 审计对象为胜平负三分类概率：以赛前留档的 `pbase` 和 `qmkt` 为输入，在按开赛时间排序的校准区间中拟合收缩权重，在更晚的验证区间评估 Brier Score、Log Loss 与校准误差。大小球和让球需要另行验证比分分布层，不能直接沿用该结论。

所有模拟信号必须同时通过：

| 校验 | 门槛 |
|---|---:|
| 近期有效比赛 | 双方各至少 5 场 |
| 统一庄家 | 取得 `Pinnacle` 的对应全场盘口 |
| 研究试算 EV | 至少 5% |
| 模型相对市场优势 | 至少 8 个百分点 |
| 保守 EV | 至少 3% |
| 胜平负分布最大模型与市场概率偏差 | 不超过 15 个百分点；超出则整场三类市场暂停 EV 展示 |
| 胜平负方向模型概率 | 至少 40% |
| `pfinal` 校准和回测验证 | 必须通过，当前尚未完成 |

任一条件失败，动作必须为 `观望` 或 `市场缺失`，资金占用为零，并输出拒绝原因。当前 API 模式可保留研究试算 EV 供诊断，但不得输出正式模拟信号；重大模型分歧时，主页面和模拟舱不显示研究 EV 数值，仅在审计报告附录保留原始试算。

## 6. 数据质量和报告

页面、Excel 与 PDF 报告至少展示：

- 官方中文赛事名和双方球队名；
- 北京时间和场地；
- 指定庄家及赔率抓取时间；
- 双方近期有效比赛覆盖数；
- 独立模型概率、市场去水概率、最终展示概率；
- EV、保守 EV、模拟动作及拒绝原因；
- 数据质量说明。
- `pfinal` 验收状态、结构化报价数量、合格赛前快照及已隔离赛后快照数量。

## 7. 回测闭环要求

进入正式验收前，必须保存赛后 90 分钟比分、每笔模拟动作、下注时赔率、结算盈亏和资金曲线，并输出：

- Brier Score；
- Log Loss；
- ROI；
- 最大回撤；
- 分市场与分赛事统计；
- 具备赔率时点后再增加 CLV 分析。

- 已隔离开赛后生成的快照/预测，不将其纳入 `pshr/pfinal` 验收。

本规范优先解决数据混入和虚假信号问题，不代表模型已经经过历史回测证明可盈利。
