# 世界杯预测项目不足分析

日期：2026-05-28  
范围：本地版“世界杯预测”项目，暂不继续网页版公网发布  
结论口径：只评估 90 分钟赛果、API-Football 数据链、Pinnacle 全场盘口、模拟舱和中文报告交付

## 1. 总体结论

当前项目已经具备“本地预测工作台”的基础交付形态：可以启动本地页面、抓取赛前比赛、保存快照、生成胜平负/大小球/让球研究结果、展示数据处理过程，并导出中文 Excel/PDF 报告。

但它还不能被定义为“已验证的正期望买入模型”。真正的核心缺口不是界面，而是模型验证样本、正式概率 `pfinal`、大小球/让球专项校准、长期回测和收盘线验证仍未完成。

一句话判断：

```text
当前可交付：本地数据抓取 + 预测展示 + 研究复核 + 报告导出系统
当前不可宣称：稳定盈利模型、正式 BUY 信号系统、真实投注决策工具
```

## 2. 当前审计到的项目状态

### 2.1 功能与测试状态

- 本地代码测试通过：`56` 项测试全部通过。
- 本地数据库记录：
  - 预测运行：`81` 条。
  - API 快照：`33` 条。
  - 结构化盘口报价：`425` 条。
  - 已同步 90 分钟赛果：`2` 场。
- API 自动预测记录中历史曾出现过资金占用记录：`auto` 模式 `45` 条，累计 `23` 个 active_bets，累计占用 `410` 元。这主要来自前期版本的研究逻辑，不能与当前正式闸门混用。

### 2.2 模型验收状态

当前模型验收状态为：

```text
状态：校准样本不足
正式 EV：关闭
pfinal：未批准
合格赛前样本：2 / 100
独立比赛数量：2 / 100
校准区间样本：0 / 20
验证区间样本：0 / 20
```

市场覆盖状态：

```text

Pinnacle 报价：425
带赔率时点快照：11
合格赛前快照：7
赛后隔离快照：4
```

这说明当前数据链已经开始形成，但离真正模型验收还有明显距离。

## 3. P0 级不足：必须先解决

### 3.1 正式 EV 仍未启用，不能输出真正 BUY 信号

项目文档和代码都明确：当前 `pbase` 只是基础模型概率，`qmkt` 是市场去水概率，`pshr` 还在等待样本验收，`pfinal` 尚未实现或批准。

现在模拟舱中的 EV 本质上仍是“研究试算 EV”，不是正式可执行 EV。即使页面出现方向，也必须理解为复核信号，而不是投注指令。

风险：

- 用户容易把“研究 EV 为正”误解为“可以买”。
- 早期历史记录里已有 active_bets，容易污染后续回测口径。
- 若不继续强化文案和数据隔离，会导致交付边界不清。

建议：

1. 页面主文案统一改为“研究复核 / 观望 / 市场缺失”，在 `pfinal` 通过前不出现“买入”字样。
2. 历史记录增加模型版本筛选，只把当前治理版本后的记录用于展示。
3. 报告首页增加醒目声明：正式 EV 未启用，模拟舱不构成正式信号。

### 3.2 合格样本只有 2 场，远低于 100 场门槛

当前最关键的短板是样本数。只有 2 场合格赛前已结算样本，无法评估：

- Brier Score 是否稳定。
- Log Loss 是否优于市场。
- `pshr` 收缩权重是否有效。
- 模型在不同联赛、赔率区间、热门/冷门方向上是否失真。

风险：

- 单场或少数比赛的 ROI 没有统计意义。
- 高赔率方向容易制造“账面正 EV”，但长期可能是模型高估冷门。
- 大小球/让球尤其容易因比分分布偏差产生夸张 EV。

建议：

1. 先完成 100 场合格赛前样本，不做正式信号解锁。
2. 每天只采集赛前、有 Pinnacle、盘口完整、开赛时间可靠的比赛。
3. 赛后自动同步 90 分钟比分，形成可回测闭环。

### 3.3 大小球和让球还没有专项验收

当前时间切分校准只覆盖胜平负三分类概率。大小球和让球虽然已经用比分矩阵处理走水、半赢、半输，但还没有验证比分矩阵本身是否可靠。

风险：

- 大小球 EV 可能继续出现异常放大。
- 让球盘如果盘口方向或盘口线解释有偏差，会直接影响收益计算。
- 胜平负校准通过也不能自动证明大小球和让球可用。

建议：

1. 为大小球建立独立指标：总进球分布校准、Over/Under Brier、盘口线分桶 ROI。
2. 为让球建立独立指标：让球线分桶、热门让球/受让方向表现、走水率。
3. 未完成前，大小球和让球只能显示“研究值”，不能计入正式资金。

## 4. P1 级不足：影响模型质量

### 4.1 基础模型仍偏 MVP，球队实力底座较弱

当前球队画像主要来自最近 5-10 场的进球、失球和积分，再映射为简化 Elo、攻防评分和阵容深度。这适合做第一版解释型模型，但不够支撑世界杯级别预测。

缺失字段：

- 国际/俱乐部 Elo 或权威强度评级。
- FIFA 排名及排名变化。
- 阵容身价、球员出勤、伤停、预计首发。
- xG、xGA、射门、射正、红牌、点球等过程数据。
- 主客场、旅行、天气、赛程密度的可靠量化。

建议：

1. 建立“基础强度评分层”，先接入 Elo/FIFA/近期状态三类特征。
2. 把“近期 10 场”从唯一核心，降级为状态修正项。
3. 对世界杯赛事单独增加国家队阵容与赛事阶段变量。

### 4.2 市场数据依赖 Pinnacle，但覆盖不稳定

项目选择固定庄家是正确的，能避免不同公司拼盘口。但 API-Football 对 Pinnacle 的赔率覆盖不稳定，导致很多真实比赛只能展示预测，无法进入模拟舱或校准样本。

风险：

- 数据采集效率低。
- 很多比赛没有盘口，无法形成合格样本。
- 无收盘赔率，无法做 CLV 验证。

建议：

1. 保持正式口径只用 Pinnacle，但把“采集候选池”扩大到有 Pinnacle 覆盖的赛事。
2. 每个 fixture 至少保存早盘、即时盘、临场盘三个时点。
3. 增加收盘线字段，后续用于 CLV。

### 4.3 中文名录仍是受控映射，不是自动官方中文库

API-Football 不提供中文官方名。当前策略是：能识别到中文就优先中文，识别不到保留英文。这是正确的，但名录仍需持续补充。

风险：

- 冷门联赛和小球队仍可能显示英文。
- 同一球队可能存在多个英文别名，影响用户识别。
- 若手工翻译不审计，可能出现误译。

建议：

1. 增加“未翻译球队/联赛清单”导出。
2. 由甲方逐批核定中文名。
3. 保存 API 原名、中文名、国家、league_id/team_id 的映射版本。

### 4.4 世界杯专属情景变量还没有真实落地

项目理论上考虑了小组积分、历史关系、国家关系、商业收益等维度，但当前多数还停留在低权重假设或未接入真实数据。

风险：

- 这些变量很容易变成故事解释，而不是可验证特征。
- 权重若手工设定，会造成主观偏差。
- 世界杯小组赛和淘汰赛的策略逻辑差异还没有明确建模。

建议：

1. 小组赛先做“出线压力函数”，不要做模糊叙事加分。
2. 国家关系和商业收益只保留为备注或敏感性假设，不进入正式模型。
3. 淘汰赛单独建模 90 分钟保守倾向和平局倾向。

## 5. P2 级不足：交付和运营层面

### 5.1 历史记录需要治理和清洗

本地数据库里有早期版本产生的历史预测记录，其中部分记录曾经有资金占用。这些记录不能和当前“正式 EV 关闭”的治理版本混在一起分析。

建议：

1. 给每条预测保存更细的模型版本、治理版本和规则版本。
2. 历史页面默认只展示当前规则版本。
3. 老版本记录归档为“历史研究记录”，不进入验收统计。

### 5.2 测试入口还不够统一

当前 `pytest` 不在系统 Python 环境中，但可以通过标准测试入口跑通 56 项测试。说明项目测试本身可用，但交付包还缺少一个普通用户可理解的“验收测试按钮/脚本”。

建议：

1. 增加 `run_tests.command`，双击即可跑基础测试。
2. 在交付说明里写清楚：功能启动、API 检查、报告导出、模型验收状态四个验收动作。
3. 将测试依赖写入开发依赖，避免换电脑后测试入口不一致。

### 5.3 公网分享线暂停后，交付边界要重新写清

现在用户决定暂时不做网页版分享，因此交付范围应回到本地电脑运行。已有公网部署配置可以保留，但不应作为当前交付重点。

建议：

1. 当前交付说明改成“本地正式版”。
2. 公网部署文档标记为“预留能力，暂不验收”。
3. API 密钥只保留在本机环境变量，不上传、不写入仓库。

## 6. 模型层面的核心问题

### 6.1 概率来源还不够“可证明”

当前基础概率来自简化评分 + 泊松比分矩阵，市场概率来自 Pinnacle 去水，展示概率是手工权重融合。这条链路可解释，但还没有证据证明它比市场更准。

下一步应从“预测方向好不好看”转为回答三个问题：

1. `pbase` 是否有独立预测价值？
2. `pbase + qmkt` 收缩后的 `pshr` 是否优于单独 `pbase`？
3. `pshr` 是否至少不显著劣于市场，并能在验证集控制回撤？

### 6.2 EV 不是单独公式问题，而是概率可信度问题

EV 公式本身并不复杂：

```text
胜平负研究 EV = pbase * odds - 1
正式 EV = pfinal * odds - 1
大小球/让球 EV = sum(比分概率 * 盘口结算收益)
```

真正的问题是：当前 `pbase` 还没有被证明是可执行概率，大小球/让球的比分分布也还没有被证明校准。因此即便公式正确，输入概率不可靠，输出 EV 也会失真。

### 6.3 高赔率冷门尤其需要强限制

前期出现“模型概率低但赔率高仍买入”的问题，本质是冷门方向的概率高估。现在已加入胜平负 40% 下限、优势门槛、保守 EV 和分歧暂停，但这仍属于工程防护，不等于模型被证明有效。

建议继续增加：

- 冷门赔率区间分桶回测。
- 模型概率 20%-35% 区间的专项校准。
- 市场分歧超过阈值时不展示 EV，只展示复核原因。

## 7. 建议的下一步执行路径

### 阶段 1：先把“不能买”的边界做硬

目标：防止误用。

动作：

1. 页面和报告统一改成“研究复核”，不显示正式买入。
2. 历史记录按治理版本隔离。
3. 模拟舱只显示资金占用为 `0`，直到 `pfinal` 通过。

验收标准：

```text
API 模式下 formalEvEnabled=false 时，任何市场都不能产生真实资金占用。
```

### 阶段 2：建立合格样本池

目标：从 2 场合格样本提升到 100 场。

动作：

1. 每天自动采集有 Pinnacle 全场盘口的赛前比赛。
2. 保存 fixture、赔率、赔率时点、近期比赛、模型概率和盘口键。
3. 赛后自动同步 90 分钟比分。

验收标准：

```text
合格赛前已结算样本 >= 100
独立比赛 >= 100
校准区间 >= 20
验证区间 >= 20
```

### 阶段 3：完成胜平负校准

目标：验证 `pshr`。

动作：

1. 按开赛时间切分开发、校准、验证。
2. 比较 `pbase`、`qmkt`、`pshr` 的 Brier Score、Log Loss、校准误差。
3. 输出分赔率区间、分联赛、分热门/冷门的表现。

验收标准：

```text
pshr 不劣于 pbase
pshr 不显著劣于 qmkt
冷门方向不过度高估
```

### 阶段 4：大小球和让球专项验收

目标：解决 EV 异常放大。

动作：

1. 对总进球分布做校准。
2. 对让球盘按盘口线分桶回测。
3. 统计全赢、半赢、走水、半输、全输比例。

验收标准：

```text
大小球和让球分别有独立回测报告
没有单一盘口线支撑整体结论
EV 异常值可以被盘口、概率或数据原因解释
```

### 阶段 5：再讨论是否启用正式 `pfinal`

目标：谨慎开放正式模拟信号。

动作：

1. 固化 `pshr -> pfinal` 公式。
2. 明确最小概率、最大赔率、最大分歧、最大回撤限制。
3. 由甲方确认是否允许从“研究复核”进入“正式模拟”。

验收标准：

```text
pfinalStatus=approved
formalEvEnabled=true
报告中出现正式 EV 的同时给出完整回测证据
```

## 8. 甲方视角的交付判断

当前项目适合继续作为：

- 本地足球比赛预测分析系统。
- 赛前数据采集和留档系统。
- 模型研究、复盘和报告生成工具。
- 后续正式模型验收的基础工程。

当前项目不适合被包装成：

- 已经能稳定赚钱的系统。
- 可以直接指导真实投注的工具。
- 完整世界杯量化交易模型。
- 已完成历史验证的正式 BUY 信号产品。

## 9. 最重要的改进原则

后续不要优先追求“更多漂亮功能”，而应优先解决这四件事：

1. 让用户永远知道当前结果是不是正式信号。
2. 让每个 EV 都能追溯到概率来源、盘口来源和结算规则。
3. 让每个模型结论都能被历史样本验证。
4. 让模型承认“不下注 / 观望”是正常且重要的结果。
